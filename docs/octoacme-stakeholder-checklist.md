## Purpose
Help project teams identify, categorize, and plan communication with all relevant stakeholders throughout the project lifecycle.

## How to Use
- Complete this checklist during project initiation
- Review and update at each major milestone
- Use to plan stakeholder-specific communications
- Reference when escalating issues or making decisions

---

## Stakeholder Identification

### Internal Stakeholders

#### Executive/Leadership
- [ ] Project Sponsor identified
- [ ] Budget/resource approver identified
- [ ] Strategic decision maker identified
- **Communication cadence**: Monthly or milestone-based
- **Key updates needed**: ROI, timeline, major risks

#### Product & Engineering Leadership
- [ ] Product Lead identified
- [ ] Engineering Lead identified
- [ ] Architecture/Technical Lead identified
- **Communication cadence**: Weekly or bi-weekly
- **Key updates needed**: Progress, technical decisions, blockers

#### Core Delivery Team
- [ ] Project Manager assigned
- [ ] Product Manager assigned
- [ ] Scrum Master assigned (if using agile)
- [ ] Development team lead identified
- [ ] UX/UI Designer assigned (if needed)
- [ ] DevOps Engineer assigned
- [ ] Technical Writer assigned (if needed)
- [ ] QA/Testing lead assigned
- **Communication cadence**: Daily standups, sprint ceremonies
- **Key updates needed**: Tasks, blockers, progress

#### Support & Operations
- [ ] Customer Support lead identified
- [ ] IT Operations contact identified
- [ ] Security team contact identified
- [ ] Compliance/Legal contact identified (if required)
- **Communication cadence**: As needed, pre-release briefings
- **Key updates needed**: Feature changes, support implications, runbooks

#### Adjacent Teams
- [ ] Dependent team leads identified
- [ ] Integration partners identified
- [ ] Shared resource owners identified
- **Communication cadence**: Weekly or as dependencies require
- **Key updates needed**: Dependencies, interface changes, timelines

### External Stakeholders

#### Customers/Users
- [ ] Beta user group identified (if applicable)
- [ ] Customer advisory board contacted (if applicable)
- [ ] User research participants recruited
- **Communication cadence**: Per research plan, release announcements
- **Key updates needed**: New features, breaking changes, migration guides

#### Partners/Vendors
- [ ] Third-party integration partners identified
- [ ] Vendor contacts for dependencies identified
- [ ] External API consumers identified
- **Communication cadence**: As changes affect them
- **Key updates needed**: API changes, deprecations, timelines

## Stakeholder Analysis Matrix

For each stakeholder or group, assess:

| Stakeholder | Power (H/M/L) | Interest (H/M/L) | Strategy | Primary Contact |
|-------------|---------------|------------------|----------|-----------------|
| [Example: Project Sponsor] | High | High | Manage Closely | PM |
| [Example: Customer Support] | Low | High | Keep Informed | Technical Writer |
| | | | | |
| | | | | |

**Strategy Guide**:
- **High Power, High Interest**: Manage Closely (frequent updates, involve in decisions)
- **High Power, Low Interest**: Keep Satisfied (periodic updates, focus on outcomes)
- **Low Power, High Interest**: Keep Informed (regular updates, address concerns)
- **Low Power, Low Interest**: Monitor (minimal updates, inform of major changes)

## Communication Plan by Stakeholder Group

### Executives/Sponsors
- **Frequency**: Monthly or at milestones
- **Method**: Executive summary, dashboard, brief meeting
- **Content**: Progress vs. plan, budget, risks, decisions needed
- **Owner**: Project Manager

### Product & Engineering Leads
- **Frequency**: Weekly
- **Method**: Status report, sync meeting
- **Content**: Detailed progress, technical decisions, resource needs, risks
- **Owner**: Project Manager + Scrum Master

### Delivery Team
- **Frequency**: Daily (standups), Sprint ceremonies
- **Method**: Standup, sprint planning, retrospective, demo
- **Content**: Task progress, blockers, upcoming work, improvements
- **Owner**: Scrum Master

### Support & Operations
- **Frequency**: Pre-release + as needed
- **Method**: Documentation, training session, Q&A
- **Content**: Feature overview, support implications, runbooks, FAQs
- **Owner**: Technical Writer + Product Manager

### Customers/Users
- **Frequency**: Release announcements, major milestones
- **Method**: Release notes, blog post, email, in-app notifications
- **Content**: New features, improvements, migration guides
- **Owner**: Product Manager + Technical Writer

## Communication Responsibilities by Role

| Role | Communication Responsibilities |
|------|-------------------------------|
| **Project Manager** | Overall project status, timeline, risks, dependencies, stakeholder coordination |
| **Product Manager** | Product vision, roadmap, feature priorities, customer impact, success metrics |
| **Scrum Master** | Team velocity, process improvements, impediments, ceremony facilitation |
| **Developers** | Technical implementation details, code reviews, architecture decisions |
| **UX/UI Designer** | Design rationale, user research findings, design reviews, accessibility |
| **DevOps Engineer** | Infrastructure changes, deployment plans, monitoring, incident response |
| **Technical Writer** | Documentation, release notes, user guides, support materials |
| **QA/Testing** | Test results, quality metrics, defect status, acceptance verification |

## Escalation Matrix

|------------|--------------|-------------------|-------------------|-------------------|
| Technical blocker | Scrum Master | Project Manager | Engineering Lead | Sponsor |
| Design/UX issue | UX/UI Designer | Product Manager | Product Lead | - |
| Infrastructure | DevOps Engineer | Platform Team Lead | Infrastructure Lead | - |
| Scope change | Project Manager | Product Manager | Product Lead | Sponsor |
| Resource constraint | Scrum Master | Project Manager | Engineering Lead | Sponsor |
| Timeline risk | Project Manager | Product Lead | Sponsor | - |

## Checklist for Effective Stakeholder Management

- [ ] All stakeholders identified and categorized
- [ ] Communication plan created for each stakeholder group
- [ ] Communication owners assigned
- [ ] Regular communication cadence established
- [ ] Escalation paths documented and shared
- [ ] Decision log template in place
- [ ] Stakeholder feedback channels established
- [ ] Communication effectiveness reviewed at retrospectives
